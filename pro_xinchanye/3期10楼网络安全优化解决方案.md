# 3期10楼仪器研发网络安全优化解决方案

---

## 一、项目背景

### 1.1 现状概述

3期10楼为仪器研发组专用办公及开发区域，当前网络与2期、3期、4期全网互通。随着研发业务复杂度提升（含超融合集群、AI应用开发、多套生产系统），10楼网络面临安全边界模糊、访问控制粒度不足等问题，亟需进行安全加固。

### 1.2 需求目标

- 对3期10楼研发网络进行安全隔离，缩小攻击面
- 精细化控制10楼与2期机房、其他区域之间的双向访问策略
- 在不新增防火墙硬件投入的前提下，最大化利用现有安全设备
- 保障研发组正常访问2期各系统（安全网关、网络AC、监控、备份等），同时不影响外部用户访问10楼业务（PLM、MES、打印机服务等）

### 2.3 现状问题诊断

1. **安全边界模糊**：10楼研发网与生产网全网互通，一旦研发环境遭受攻击，可横向扩散至全区域
2. **访问控制粗放**：当前仅依赖VLAN做基本隔离，缺乏基于IP/端口/协议的精细化策略控制
3. **合规风险**：仪器研发涉及知识产权保护，与生产/办公混网存在数据泄露隐患
4. **设备复用不足**：4期已启用但业务量小、启用楼层不多，山石防火墙性能利用率低，可充分挖掘其策略控制能力为3期10楼服务
5. **360杀毒未深度利用**：全区域虽已部署360天擎，但仅使用基础查杀功能，终端检测响应(EDR)、威胁狩猎、资产清点、漏洞管理等高阶能力未启用；10楼隔离后需确保研发终端管控策略独立配置、病毒库更新通道畅通
6. **Ip_guard策略粗放**：10楼研发组已部署Ip_guard客户端，但DLP策略未针对研发场景精细化（如源代码防泄漏、AI模型数据保护、外发审批流程），且策略与网络隔离方案需协同设计

## 三、需求可行性分析

### 3.1 方案一（推荐）：利用4期防火墙 + VLAN隔离 + 策略路由引流

**技术可行性：★★★★★（高）**

#### 核心思路

**防火墙物理位置不动**，通过在3期汇聚交换机上配置策略路由（PBR），将10楼研发VLAN的流量**下一跳指向4期山石防火墙**，由4期防火墙执行安全策略检查后，再将合法流量回注正常路径到达目的地。实现"流量走过去做策略，设备不挪窝"。

```
逻辑示意:

  10楼研发流量 ──→ 3期汇聚交换机 ──[PBR]──→ 4期山石防火墙 ──[策略检查]──→ 2期/Internet
                   (下一跳改写)            (物理位置不动)
```

#### 优势

- **零硬件成本**：复用4期低负载的山石防火墙（4期当前启用楼层少、业务量小，防火墙性能充足），仅涉及配置变更
- **零物理施工**：不动机柜、不拉线、不断电，纯网络配置变更
- **不影响4期现有业务**：4期防火墙自身性能利用率低，10楼策略仅消耗空闲性能；未来4期业务增长需回收时，调整PBR下一跳即可，无需反向迁移
- **技术成熟**：PBR+防火墙策略隔离是业界成熟方案，华为交换机原生支持策略路由
- **可扩展**：后续可按需将更多VLAN流量引入防火墙，实现渐进式安全加固

#### 需解决的问题及应对

| 问题 | 应对方案 |
|------|----------|
| 10楼研发组需访问2期安全网关/AC/监控/备份等系统 | 4期防火墙上配置单向放行策略：10楼→2期指定IP+端口白名单 |
| 外部用户需访问10楼PLM/MES/打印机服务 | 4期防火墙上配置反向放行策略：特定源IP→10楼服务器指定端口；PBR需配置双向（出站+入站）引流 |
| 10楼超融合集群仅供内部使用 | 默认拒绝外部访问超融合网段，仅开放必要管理端口 |
| PBR增加网络跳数，可能引入延迟 | 3期↔4期间为光纤互联，额外延迟≤1ms，对业务无感知 |
| 防火墙可能成为性能瓶颈 | 评估山石防火墙吞吐量是否满足10楼流量；备份等大流量通过ACL直通不引向防火墙 |
| DNS服务在2期，10楼需解析 | 放行10楼→2期DNS（UDP 53），或考虑在3期部署DNS缓存 |

### 3.2 方案二（参考）：新增防火墙 + 独立ADSL

**技术可行性：★★（低）**

#### 客户反馈

- **成本高**：需新购防火墙设备、ADSL线路月租、施工费用
- **技术不可行**：独立ADSL意味着10楼完全脱离现有网络，但20+套业务系统（PLM/MES/监控等）部署在10楼服务器上供全区域使用，独立后其他区域无法访问

#### 结论

此方案**不建议采纳**，但可考虑作为远期规划——当研发业务完全独立（服务器迁出10楼）后再行评估。

## 四、推荐方案详细设计

### 4.1 网络改造总体设计

**核心变更：在3期汇聚交换机上新增策略路由（PBR），将10楼VLAN流量引向4期防火墙执行策略检查。**

物理拓扑不变，仅涉及：

| 变更点 | 设备 | 变更内容 |
|--------|------|----------|
| PBR策略路由 | 3期汇聚交换机 | 新增：匹配10楼VLAN的流量，下一跳指向4期山石防火墙 |
| 回程路由 | 4期核心交换机 | 新增：防火墙回程流量指向2期/3期的路由 |
| 安全策略 | 4期山石防火墙 | 新增：10楼安全域策略（见4.4策略表） |
| VLAN划分 | 3期汇聚/10楼接入 | 新增：10楼按用途划分VLAN（见4.2） |

### 4.2 VLAN规划

| VLAN ID | 网段 | 用途 | 安全级别 |
|---------|------|------|----------|
| VLAN 100 | 10.0.100.0/24 | 10楼研发PC | 高（隔离区） |
| VLAN 110 | 10.0.110.0/24 | 10楼超融合集群（内部） | 高（仅研发访问） |
| VLAN 120 | 10.0.120.0/24 | 10楼物理服务器（PLM/MES/打印机/监控等） | 中（需对全区域开放特定端口） |
| VLAN 200 | 10.0.200.0/24 | 10楼管理网段（带外管理） | 最高（仅IT运维） |
| VLAN 300 | 10.0.10.0/24 | 3期其他楼层办公网 | 低（已有） |

> **说明**：将10楼服务器（PLM/MES/打印机等）单独划分VLAN 120，与研发VLAN 100/110分离——实现"研发内部隔离"的同时，便于对外部开放服务器访问。

### 4.3 防火墙部署模式：PBR引流模式

**防火墙物理位置不变，部署在4期核心交换机侧。** 通过在3期汇聚交换机上配置策略路由（PBR），将10楼VLAN流量引向4期防火墙。

#### 部署架构

```
                        4期机房
                   ┌─────────────────┐
                   │  4期 核心交换机    │

                   │  ┌────▼──────┐   │
                   │  │ 山石防火墙  │   │
                   │  │            │   │
                   │  │ Trust接口:  │   │ ← 连接4期核心交换机
                   │  │ 接收PBR引流 │   │   作为被引流流量的入口
                   │  │            │   │
                   │  │ Untrust:   │   │ ← 策略放行后回注4期核心
                   │  │ 回程接口    │   │
                   │  └───────────┘   │
                   └─────────────────┘
                           │ 光纤
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
    ┌────▼────┐      ┌─────▼─────┐      ┌────▼────┐

                     └─────┬─────┘
                     ┌─────▼─────┐
                     │ 10楼接入   │
                     └───────────┘
```

#### PBR引流设计要点

| 设计要素 | 配置 |
|----------|------|
| **PBR匹配条件** | 源IP ∈ 10楼VLAN网段（VLAN 100/110/120）的流量 |
| **PBR动作** | 设置下一跳为4期山石防火墙Trust接口IP |
| **排除项** | 10楼VLAN间互访（VLAN 100/110 ↔ VLAN 120）不走PBR，本地直通 |
| **防火墙接口** | Trust Zone接4期核心（接收引流流量），Untrust Zone回程（策略放行后流量返回核心） |
| **NAT** | 不做SNAT，保持源IP透明，便于审计追溯 |
| **路由回程** | 4期核心交换机需有指向2期/3期的回程路由；或防火墙以透明模式桥接（备选） |

#### 模式对比

| 维度 | PBR引流（推荐） | 物理串联部署 | 透明桥接 |
|------|:--:|:--:|:--:|
| 物理施工 | ✅ 无 | ❌ 需机柜空间+布线 | ❌ 需机柜空间+布线 |
| 网络割接复杂度 | 低（仅PBR配置） | 高（断网割接） | 中 |
| 故障切换 | 撤销PBR即恢复 | 需Bypass硬件 | 支持Bypass |
| 4期业务增长后回收 | 改PBR下一跳即可 | 需反向迁移 | 需反向迁移 |
| 额外延迟 | ≤1ms（光纤直连） | 0 | 0 |
| 推荐场景 | ✅ 本次推荐 | 新大楼独立部署 | 对延迟极度敏感 |

### 4.4 防火墙安全策略设计

#### 4.4.1 策略总则：默认拒绝 + 白名单放行

```
4期防火墙按安全域划分:
  Trust Zone:  连接4期核心交换机，接收PBR引流的10楼流量
  Untrust Zone: 连接4期核心交换机，策略放行后回程

默认策略：Any → Any : DENY（记录日志）
在此基础上逐条开放业务所需的最小权限
```

### 4.5 PBR策略路由方案（核心）

这是整个方案的**核心机制**——通过策略路由将10楼流量"牵引"到4期防火墙，实现安全策略检查。

#### 4.5.1 PBR配置逻辑（3期汇聚交换机）

```
PBR策略配置（华为交换机，实际命令以现场设备型号及OS版本为准）:

acl advanced 3010
 rule 5  permit ip source 10.0.100.0 0.0.0.255   # VLAN 100 研发PC
 rule 10 permit ip source 10.0.110.0 0.0.0.255   # VLAN 110 超融合

acl advanced 3020
 rule 5  permit ip destination 10.0.120.0 0.0.0.255  # 目的为VLAN 120 服务器

policy-based-route PBR_10F_TO_FW permit node 10
 if-match acl 3010                                 # 匹配10楼出站
 apply next-hop <4期防火墙Trust接口IP>              # 下一跳→4期防火墙

policy-based-route PBR_TO_10F_SERVER permit node 10
 if-match acl 3020                                 # 匹配入站到10楼服务器
 apply next-hop <4期防火墙Trust接口IP>              # 下一跳→4期防火墙
```

#### 4.5.2 PBR排除项（不引向防火墙的流量）

以下流量在PBR中**显式排除**，本地直通，避免增加防火墙负载：

| 排除项 | 原因 | 配置方式 |
|--------|------|----------|
| 10楼VLAN间互访（VLAN 100/110 ↔ VLAN 120） | IP_guard服务端也在3期；研发访问PLM等10楼本地服务器无需经过防火墙 | ACL deny节点在PBR permit之前 |
| 备份流量（2期 ↔ 3期10楼备份服务器） | 大流量（TB级），经过防火墙会拖垮性能 | ACL匹配备份服务器IP+端口，PBR拒绝匹配 |
| 360客户端→Ip_guard服务端（同楼） | 管控通信本地可达，无需出楼 | 目的IP=3期本地，PBR不匹配 |
| 超融合集群东西向流量 | VLAN 110内部L2通信，不经过3期汇聚 | 无需处理 |

#### 4.5.3 故障切换机制

```
PBR故障切换（利用交换机的track联动物理状态检测）:

  PBR node 10
    if-match acl 3010
    apply next-hop <4期防火墙IP> track 1    # 关联track

  track 1
    type nqa icmp-echo destination <4期防火墙IP>
      next 3期汇聚→4期光纤链路状态 + 防火墙可达性

  当track 1检测失败（防火墙不可达/光纤中断）:
    → PBR node 10 自动失效
    → 流量回退到正常路由表
    → 10楼网络恢复未隔离状态（Fail-Open，保证业务连续性）
    → 触发告警通知IT运维
```

#### 4.5.4 分层控制架构

```
            ┌──────────────────────────────────────┐
            │         第一层：3期汇聚交换机           │
            │                                      │
            │  ACL: 10楼VLAN ↔ 其他楼层VLAN 粗粒度隔离 │
            │  PBR: 10楼VLAN出/入站 → 重定向至4期防火墙 │
            │  PBR排除: 备份/本地互访 直通             │
            └──────────────────┬───────────────────┘
                               │ 光纤（PBR流量）
                               ▼
            ┌──────────────────────────────────────┐
            │         第二层：4期山石防火墙           │
            │                                      │
            │  L3/L4 ACL: IP+端口白名单              │
            │  IPS/AV: 入侵防御+防病毒                │
            │  会话日志: 全量记录                     │
            └──────────────────┬───────────────────┘
                               │ 策略放行后回程
                               ▼
            ┌──────────────────────────────────────┐
            │         第三层：2期山石防火墙(出口)      │

```

> **说明**：三层防护中，第一层做流量牵引+粗粒度ACL，第二层做精细化策略+威胁检测（本次新增），第三层做出口防护（已有）。PBR是连接第一层和第二层的"管道"。

### 4.6 备份流量优化建议

**问题**：2期机房数据备份至3期10楼机房，备份数据量大（TB级），如果经PBR引向4期防火墙会严重拖垮防火墙性能。

**建议方案**：

- 在3期汇聚交换机的PBR策略中，通过ACL**显式排除备份流量**，使其不经4期防火墙
- 备份源/目的IP匹配后，PBR不命中，走正常路由表直达
- 备份流量与业务流量在网络层逻辑隔离，备份VLAN（建议VLAN 999）仅备份服务器间可达

```
PBR排除逻辑（3期汇聚交换机）:

  ACL备份排除:
    permit ip source <2期备份源IP> destination <3期10楼备份目标IP>
    permit ip source <3期10楼备份目标IP> destination <2期备份源IP>

  PBR node 5 (在引流node之前):
    if-match 备份ACL → permit (不设next-hop，走正常路由)
    if-match 10楼VLAN ACL → apply next-hop 4期防火墙

备份流量路径:
  2期备份源 → 2期核心 → 光纤 → 3期汇聚(PBR不命中，直通) → 10楼备份目标
```

> **远期建议**：为备份流量建设独立光纤链路或专用备份网络，彻底分离生产与备份流量。

### 4.8 网络准入控制（NAC）方案

VLAN隔离 + PBR + 防火墙策略解决了**合法IP的访问控制**问题，但无法防止**非法设备接入**或**不合规终端混入**研发网段。网络准入控制（NAC）补上这最后一块短板。

#### 4.8.1 NAC要解决的问题

```
场景1: 未知设备私接
  外来人员笔记本插入10楼空闲网口 → DHCP获取VLAN 100的IP
  → 直接进入研发网段，VLAN隔离和防火墙策略对它无效

场景2: 非标研发设备
  Raspberry Pi / 开发板 / 测试设备直接插网线
  → 无360杀毒、无Ip_guard、无AD加域 → 裸设备在研发VLAN中

场景3: 终端合规无强制
  策略要求"研发PC必须装360+Ip_guard+打补丁才能联网"
  → 目前靠制度约束，终端不合规也能上网

场景4: 访客/外包厂商
  仪器调试工程师到现场，需临时接入网络调试设备
  → 只能给研发网IP，权限收不回来，事后忘记回收
```

#### 4.8.2 现有NAC基础盘点

客户的现有基础设施已经具备启用NAC的基本条件：

| 已有资产 | 部署位置 | NAC相关能力 |
|----------|----------|------------|
| **深信服全网行为管理** | 2期机房 | 通常含准入控制模块（Portal/802.1X认证 + 终端合规检查 + 访客管理） |
| **网络AC** | 2期机房 | 如果指"网络准入控制器（NAC）"，就是策略执行的核心组件 |
| **AD域服务器** | 2期机房 | 用户身份源，NAC认证的后端 |
| **华为核心/汇聚交换机** | 2期/3期 | 支持802.1X、MAC认证、策略联动（VLAN动态分配） |
| **360天擎** | 2期机房 | 终端合规状态（杀毒运行/病毒库版本/补丁状态）可作为NAC合规检查输入 |

> **关键确认项**：实施前需确认深信服全网行为管理是否已含准入控制License、网络AC的具体产品型号和能力边界。如果已有准入模块，**近乎零成本启用**。

#### 4.8.3 NAC部署架构


#### 4.8.4 准入策略：按设备身份动态分配VLAN

| 设备类型 | 认证方式 | 合规检查 | 分配VLAN | 访问权限 |
|----------|----------|----------|----------|----------|
| **公司研发PC** | 802.1X（AD账号+机器证书） | 360运行+病毒库≤3天+补丁合规+Ip_guard运行 | VLAN 100 | 正常访问（经PBR→4期防火墙策略） |
| **公司研发PC（不合规）** | 同上 | 任一检查项不通过 | 隔离VLAN 910 | 仅可访问360更新/AD域控/补丁服务器，修复后自动切回VLAN 100 |
| **已知测试设备** | MAC认证白名单 | 免检（无360/Ip_guard） | VLAN 500(物联网) | 仅可访问指定调试服务器，禁止访问Internet/2期内网 |
| **访客/外包厂商** | Portal认证（手机号+工单号） | 免检 | VLAN 900(Guest) | 仅可访问Internet，完全隔离内网 |
| **未知设备** | MAC不在白名单 | — | VLAN 999(黑洞) | 无任何访问权限，记录告警 |

#### 4.8.5 与现有方案的协同设计

```
协同点1: NAC VLAN × PBR 防火墙策略

  NAC动态分配的VLAN自动纳入PBR管控:

  VLAN 100(合规研发PC) ──[PBR]──→ 4期防火墙 ──→ 正常策略放行
  VLAN 500(测试设备)   ──[PBR]──→ 4期防火墙 ──→ 仅放行调试服务器
  VLAN 900(Guest)     ──[PBR排除]──→ 直通Internet（不经4期防火墙，减少负载）
  VLAN 910(隔离修复)  ──[PBR]──→ 4期防火墙 ──→ 仅放行360/AD/补丁
  VLAN 999(黑洞)      无路由，彻底隔离


协同点2: NAC × 360天擎联动

  360天擎开放API → NAC策略引擎查询终端合规状态:
  - 360服务是否运行
  - 病毒库版本是否≤3天
  - 最后全盘扫描时间是否≤7天
  - 是否有未处理的高危漏洞

  不合规 → NAC强制将终端踢到VLAN 910 → 360修复 → 合规
  → NAC检测到合规状态恢复 → 自动切回VLAN 100


协同点3: NAC × Ip_guard联动

  - NAC检测到Ip_guard客户端未运行/被卸载 → 踢到修复VLAN 910
  - NAC检测到设备为开发板/测试设备（无Ip_guard） → MAC白名单+降权VLAN 500


协同点4: NAC × AD域联动

  - 用户登录AD域 → NAC获取用户组信息
  - 研发组用户 → VLAN 100
  - 员工离职/转岗 → AD账号禁用 → NAC下次认证拒绝 → 设备自动从VLAN 100移除
  - 访客Portal → AD无账号 → 短信/二维码认证 → VLAN 900
```

#### 4.8.6 NAC实施分期

```
P0（网络改造同步，利用现有设备）
  ├─ 盘点深信服行为管理/网络AC是否已有NAC模块及License
  ├─ 确认华为接入交换机802.1X支持情况
  └─ 输出NAC启用评估报告（是否需新增硬件/License）

P1（PBR上线后2-4周）
  ├─ 10楼接入交换机启用802.1X（VLAN 100试点）
  ├─ NAC策略引擎配置：AD域认证 + 研发组用户 → VLAN 100
  ├─ MAC白名单建立（已知测试设备/打印机/IP电话）
  └─ 观察模式运行（不踢人，仅记录不合规设备）

P2（PBR上线后1-2月）
  ├─ 联动360天擎：终端合规检查上线（杀毒+补丁+病毒库）
  ├─ 不合规终端 → VLAN 910隔离修复
  ├─ 访客Portal上线（VLAN 900）
  └─ MAC白名单+物联网VLAN（测试设备→VLAN 500）

P3（持续优化）
  ├─ IoT设备指纹识别 → 自动分类（无需手动维护MAC白名单）
  ├─ NAC × PBR × 防火墙策略全联动（设备类别变更 → 自动调整防火墙规则）
  └─ NAC日志对接态势感知（设备接入异常告警）
```

| **阶段六：持续运营** | 360 EDR+防火墙日志联动（态势感知对接）；Ip_guard全量加密覆盖+审计日志对接态势感知；PBR策略定期审计（确保排除项没有被绕过）；漏洞管理常态化运营 | 持续 | 低 |

### 5.2 关键注意事项

1. **业务连续性**：所有变更在非业务窗口期（周末/凌晨）执行，准备完善PBR回退方案（删除PBR配置即可恢复原状，回退时间<5分钟）
2. **策略审计**：初期将4期防火墙默认拒绝策略设为"仅记录不阻断"（Log模式），观察1-2周确认无误后再切换为Deny
3. **PBR逐网段上线**：先以VLAN 100（研发PC）单个网段试点PBR引流，验证业务正常后再逐步加入VLAN 110/120
4. **备份流量**：PBR上线前确认备份流量ACL排除规则正确，备份窗口期验证备份任务不受影响
5. **ARP/DNS**：VLAN切割+PBR变更后确认ARP表和DNS解析正常；PBR下一跳为IP地址，确认4期防火墙ARP可达
6. **360终端在线率**：PBR上线后第一时间验证10楼所有研发PC的360客户端在线状态（经PBR→4期防火墙→2期天擎），确保病毒库更新通道畅通
7. **Ip_guard策略衔接**：同楼通信不受PBR影响（PBR排除），但需确认IT远程管理Ip_guard控制台的路径畅通
8. **防火墙+终端联动**：EDR启用初期，对编译/调试工具的误报做好白名单预案，避免终端的主动防御误杀开发工具
9. **PBR性能监控**：3期汇聚交换机PBR为硬件转发（ASIC/TCAM），性能影响可忽略；但需确认TCAM表项空间充足

### 5.3 回退方案

- **PBR快速回退**：删除3期汇聚交换机上PBR配置即可恢复原状，回退时间<5分钟；所有配置变更前均保存交换机running-config备份
- **防火墙策略回退**：4期防火墙策略上线前保留配置备份，异常时可逐条回滚
- **自动Fail-Open**：PBR关联track检测（4期防火墙可达性），防火墙不可达时PBR自动失效，流量回退正常路由，保证业务连续性
- **VLAN回退**：VLAN变更前备份交换机配置，紧急情况可恢复原VLAN配置

## 七、远期建议

> **说明**：本章聚焦网络架构层面的远期优化建议。安全产品的系统性补强规划（堡垒机/NTA/蜜罐/MFA/零信任等15+产品分三期建设路线）详见**第九章**，MSS托管安全运营方案详见**第十章**。

1. **4期防火墙更新**：当前4期防火墙负载低，可满足3期10楼策略需求。建议在4期业务增长（启用楼层增多、业务量上升）时将新防火墙采购纳入预算——届时只需在3期汇聚交换机上修改PBR下一跳指向新防火墙即可无缝切换，旧防火墙留在4期继续使用或作为3期冗余
2. **PBR高可用增强**：当前PBR故障切换依赖track检测+自动Fail-Open，远期可考虑在3期汇聚交换机上配置PBR双下一跳（4期防火墙1主+4期防火墙2备），实现防火墙层面的高可用
3. **独立ADSL评估**：待10楼服务器业务（PLM/MES等）迁移至数据中心后，可重新评估10楼独立出口方案
4. **零信任架构**：结合深信服全网行为管理和态势感知，逐步向零信任架构演进（微分段 + 持续认证 + 动态授权）
5. **超融合安全加固**：超融合集群内部建议启用微分段策略，实现虚拟机级别的东西向流量控制
6. **备份链路优化**：长期来看，建议为备份流量建设独立光纤链路或专用备份网络，彻底分离生产与备份流量
7. **360 EDR ←→ 态势感知深度联动**：360天擎EDR告警与深信服态势感知平台对接，实现终端侧+网络侧攻击链关联分析；利用深信服全网行为管理的终端认证信息，将360告警定位到具体用户身份
8. **Ip_guard ←→ 零信任aTrust 动态DLP**：结合深信服零信任aTrust，实现基于访问上下文（用户角色+设备状态+网络位置）的动态DLP策略下发——研发人员在10楼内网时USB管控较宽松，远程VPN接入时自动收紧剪贴板/截屏/外发管控
9. **终端安全能力常态化运营**：建立终端安全巡检制度（360在线率、病毒库版本、漏洞修复率周报），将DLP审计日志纳入SOC日常监控，定期（每季度）复盘DLP策略有效性并迭代优化

### 9.2 一期：补盲（建议 0-6 个月）

堵住当前环境中**有漏洞但完全没覆盖**的领域：

#### 9.2.1 堡垒机（运维审计/PAM）

```
解决的问题:
  - 10楼超融合集群 + 物理服务器 root/Administrator 多人共用，操作无法追溯到人
  - 研发人员直连 SSH/RDP，误操作/恶意操作无记录无回放
  - 第三方厂商远程运维（超融合/存储/数据库）无管控窗口

部署建议:
  部署位置: 3期汇聚层（同楼低延迟），作为服务器运维统一入口
  覆盖范围: 超融合管理面 + VLAN 120 物理服务器 + VLAN 200 网络设备
  核心功能: 会话代理 + 录屏/命令记录 + 操作回放 + 高危指令拦截 + 双因子认证
  与PBR联动: 4期防火墙仅允许堡垒机IP访问服务器SSH/RDP端口，其余源IP一律拒绝
```

#### 9.2.2 数据库审计

```
解决的问题:
  - PLM/MES等系统的数据库查询/导出/删改无审计记录
  - SQL注入攻击拖库后无感知
  - 合规要求（等保/ISO27001）需数据库操作留痕

部署建议:
  部署方式: 旁路部署在3期核心交换机侧，端口镜像引流数据库流量
  覆盖范围: PLM库、MES库、监控库、超融合上的研发测试库
  告警规则: 大表导出、非工作时间批量查询、敏感列（图纸/算法字段）被访问
```

#### 9.2.3 漏洞扫描器

```
解决的问题:
  - 360 EDR管终端漏洞，但超融合虚机模板/Docker镜像的OS/中间件漏洞无人扫
  - VLAN 120对外开放的PLM/MES Web面缺乏周期性漏洞评估

部署建议:
  部署位置: 3期，扫描目标覆盖 VLAN 100/110/120/200
  周期性: 系统漏洞月扫、Web漏洞周扫、新增系统上线前必扫
  联动: 高危漏洞IP通过API自动加入4期防火墙临时阻断策略
```

#### 9.2.4 日志审计/SIEM（态势感知到位前的过渡）

```
解决的问题:
  - 防火墙/WAF/交换机/服务器/360天擎日志分散，出事后拼攻击链效率低
  - 深信服态势感知计划但尚未到位，当前无统一日志平台

部署建议:
  - 可先部署轻量级日志审计平台，或加速态势感知采购上线
  - 日志源: 4期防火墙 + WAF + 交换机syslog + AD域控 + 关键服务器
  - 至少满足: 6个月存储 + 关键字检索 + 基础关联告警
```

### 9.3 二期：增强（建议 6-12 个月）

边界已控制住，但**假设攻击者已经进来了**，这些能力帮你发现他：

#### 9.3.1 网络流量分析（NTA/NDR）

```
解决的问题:
  - 4期防火墙只看PBR引流的南北向流量，VLAN间东西向是盲区
  - 攻击者突破一台PC后横向扫描超融合集群无感知
  - 加密流量中的C2通信（DNS/ICMP隧道）防火墙未必能检测

部署建议:
  部署方式: 3期汇聚交换机端口镜像 → 旁路NTA分析
  核心能力: 流量基线学习→偏离告警、横向移动检测、加密流量威胁检测
  远期: 对接态势感知，丰富网络侧告警维度
```

#### 9.3.2 蜜罐/欺骗防御

```
解决的问题:
  - 攻击者进入内网后扫描探测，碰了蜜罐就能在损害发生前发现他
  - 研发环境特别适合蜜罐——攻击者就是冲着代码和模型来的

部署建议:
  诱饵类型:
  - 伪代码仓库（Git/SVN蜜罐，含假源码文件）
  - 伪AI模型文件（.pth/.onnx/.pb蜜罐，含水印标记）
  - 伪设计图纸共享目录（SMB/FTP蜜罐）

  部署位置: VLAN 100/120各放置2-3个轻量蜜罐
  告警策略: 正常业务不会触碰蜜罐 → 触发即P0告警
```

#### 9.3.3 APT检测/沙箱

```
解决的问题:
  - 0-day漏洞利用文档（伪装成项目需求的Word/PDF/压缩包）
  - 供应链攻击（编译器/开源库被植入后门）
  - 免杀木马（专门绕过360特征库）

部署建议:
  - 串联或旁挂在2期出口或4期防火墙侧
  - 邮件附件、Web下载文件自动送入沙箱引爆分析
  - 也可利用山石防火墙的沙箱联动能力（如已有License）
```

#### 9.3.4 主机/虚拟机微隔离

```
解决的问题:
  - VLAN内部没有隔离，同一VLAN的PC可以直接互访
  - 一台研发PC中招 → 同VLAN其他PC裸奔

部署建议:
  Option A: 360天擎微隔离模块（如果已采购License，零成本启用）
  Option B: 超融合集群分布式防火墙（对虚机东西向流量做L4策略）
  
  基础规则: 研发PC之间默认禁止SMB/RDP/SSH，仅允许必要协作端口
```

### 9.4 三期：体系化（建议 12-24 个月）

从"被动防御"走向"主动免疫"：

#### 9.4.1 多因素认证（MFA）

```
解决的问题: 研发人员密码泄露/钓鱼 → 攻击者可登入AD/VPN直进内网

覆盖范围: VPN登录 + 服务器管理员 + 堡垒机 + 超融合管理台
实现方式: 手机APP动态口令/企业微信扫码（研发区优先软件方案，降成本）
与AD集成: 10楼研发组优先启用MFA
```

#### 9.4.2 零信任架构（aTrust）

```
解决的问题: VPN接入后默认信任内网，但接入设备可能已被控
零信任理念: 不信任网络位置，每次访问都验证身份+设备+行为

部署路径:
  Step 1: 深信服行为管理 + 态势感知 → 终端信任评分
  Step 2: aTrust 替换传统VPN → 按应用授权（非按网段授权）
  Step 3: 结合Ip_guard → 动态DLP（远程接入时自动收紧管控）

已有产品联动:
  深信服aTrust ← 行为管理(终端合规) ← 态势感知(风险评估) ← Ip_guard(DLP策略)
```

#### 9.4.3 CWPP（云工作负载保护）

```
解决的问题: 超融合集群承载AI应用开发，虚机/容器层安全空白

覆盖: 虚机OS漏洞/配置基线 + Docker/K8s镜像扫描 + 运行时异常检测
建议: 确认华为FusionCube是否内置安全模块；若无，选配第三方CWPP
```

#### 9.4.4 数据安全平台 / 数据分类分级

```
解决的问题: Ip_guard管终端DLP，但服务器侧数据存储/流转没有摸底

部署: 旁路扫描VLAN 120服务器文件存储，自动识别含敏感内容的文件
联动: 发现新敏感数据源 → 自动纳入Ip_guard DLP保护范围
对数据库做敏感字段发现（如检测图纸/源码表字段）
```

#### 9.4.5 SOAR（安全编排自动化响应）

```
解决的问题: 产品多了，告警量大、人手有限、响应不过来

典型剧本:
  "EDR检测到勒索软件行为" →
    SOAR自动: ①调防火墙API隔离该IP
             ②调360天擎API对该主机深度扫描
             ③调堡垒机查该主机近24h运维操作
             ④通知安全运维群
```

#### 9.4.6 威胁情报

```
解决的问题: 只知道有人在敲门，不知道敲门的人是谁

实施: 商用威胁情报（微步/360/天际友盟等）→对接态势感知
联动: 威胁情报拉黑IP自动下发至山石防火墙
行业情报: 关注仪器制造/研发行业的定向攻击情报
```

#### 9.4.7 安全服务：渗透测试 + 红蓝对抗 + 安全培训

```
解决的问题: 技术产品永远有盲区（策略配错、人的操作绕过、配置漂移）

实施:
  - 半年度渗透测试: 模拟从Internet攻击10楼研发网
  - 年度红蓝对抗: 模拟内部威胁（被钓鱼终端为起点横向移动）
  - 安全意识培训: 研发人员钓鱼邮件演练（月/季度）
```

## 十、MSS托管安全运营方案

### 10.1 为什么需要MSS

安全建设越完善，产品越多，越需要回答一个问题：**谁来运维？**

```
产品数量视角（当前+计划共15+）:

  已有                         计划新增
  ├── 2期山石防火墙             ├── 堡垒机
  ├── 4期山石防火墙(PBR)        ├── 数据库审计
  ├── WAF                       ├── 漏洞扫描器
  ├── 深信服全网行为管理         ├── 日志审计/SIEM
  ├── 深信服态势感知(计划)      ├── NTA/NDR
  ├── 360天擎                   ├── 蜜罐
  ├── Ip_guard                  ├── APT/沙箱
  └── 华为交换机/ACL/PBR        ├── MFA / 零信任
                                ├── CWPP
                                ├── 数据安全平台
                                ├── SOAR
                                └── 威胁情报

  保守估计: 15+产品，日均数千条原始告警
  需要: 7×24值班 + 告警研判 + 事件处置 + 威胁狩猎 + 策略调优
  自建SOC团队: 3-5人起步，年成本150-300万
```

**现实矛盾：**

- 仪器制造企业 → IT团队核心任务是保障PLM/MES/ERP等生产系统 → 安全是"兼职"
- 10楼隔离 + NAC + DLP策略上线后，需持续调优 → 不是一次性项目
- 安全事件发生在凌晨/周末 → 没人值班 = 事件发现延迟 = 损害扩大

### 10.2 三种MSS运营模式

#### 模式A：远程监控 + 告警通知（轻量级）

```
MSS侧职责:
  ├── 7×24 日志采集 + 实时监控
  ├── 告警初筛（去误报 + 降噪 + 分级）
  ├── P0/P1告警 15分钟内电话/短信通知客户
  ├── 日报/周报/月报（告警趋势、TOP风险）
  └── 策略定期健康巡检

客户侧职责:
  ├── P0/P1告警接收 + 处置
  ├── 策略变更决策
  └── 安全制度执行

日志接入:
  客户侧安全设备日志 → 加密隧道 → MSS平台(SIEM/SOAR)
  → 安全分析师7×24轮值监控

适用场景: 第一次建安全运营能力，先解决"有人看"的问题
参考厂商: 深信服MSS、奇安信MSS、绿盟MSS、安恒MSS等
```

#### 模式B：远程监控 + 事件处置（标准级）

```
在模式A基础上增加:

MSS侧职责:
  ├── P0/P1告警远程处置（登录防火墙/EDR执行封禁/隔离）
  ├── 病毒/挖矿/勒索等常见事件全流程闭环
  ├── 漏洞跟踪 + 修复验证（漏洞扫描→工单→修复→复扫→闭环）
  ├── 策略持续优化（防火墙策略/EDR规则/DLP规则月度Review）
  ├── 安全态势季度分析 + 改进建议
  └── 应急响应（重大事件30分钟内启动远程处置）

客户侧职责:
  ├── 处置授权审批
  ├── 重大变更决策
  └── 安全制度执行

适用场景: 已有一定安全建设基础，希望MSS承担大部分日常运营
```

#### 模式C：驻场 + 远程结合（重载级）

```
在模式B基础上增加:

MSS侧职责:
  ├── 1名安全工程师驻场（工作日）
  ├── 远程二线专家支持（7×24）
  ├── 主动威胁狩猎（在环境中找未被发现的威胁）
  ├── 半年度红蓝对抗/渗透测试
  ├── 月度安全基线核查（设备配置/系统基线/策略审计）
  ├── 参加客户安全例会 + 年度汇报
  └── 合规审计支撑（等保/ISO27001）

适用场景: 安全要求高（核心研发数据保护），或需配合合规审计
```

### 10.3 对客户的特别契合点

```
1. 深信服生态优势
   - 深信服态势感知(计划) → 可直连深信服MSS平台，接入成本最低
   - 全网行为管理 + aTrust(远期) → 同一厂商，运营经验积累深
   - 如选深信服MSS，对自己产品的告警研判准确率更高

2. 多期互联架构复杂
   - 2期核心 + 3期汇聚 + 4期低负载 → 跨区域策略管理容易出错
   - MSS统一监控各期设备日志，提供全局视角

3. 10楼隔离后的持续运营
   - PBR策略的误拦截排查（哪些正常业务被4期防火墙误拦）
   - 防火墙策略的定期审计（哪些端口开着但实际已不需要）
   - 这些是持续性工作，不是一次性项目

4. DLP告警需人工研判
   - Ip_guard告警"研发员导出500MB代码到USB"
   - 是正常调试？还是数据窃取？→ 需要人工判断 + 与研发组沟通核实
   - MSS提供初筛+分类，减少IT团队研判负担

5. 成本优势
   - 模式A: 自建SOC的 1/4 - 1/3
   - 模式B: 自建SOC的 1/2
   - 模式C: 自建SOC的 2/3（但有驻场+专家+服务）
```
